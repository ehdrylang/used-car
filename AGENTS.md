# AGENTS.md — AI 협업 개발 규칙

이 파일은 GitHub Copilot(AI Agent)과 개발자가 협업할 때 반드시 따라야 할 규칙과 워크플로우를 정의합니다.

---

## 핵심 원칙

1. **코드는 명시적 구현 승인 없이 절대 작성하지 않는다.**
2. **항상 최소한의 변경으로 구현한다.** 기존 코드와 구조를 최대한 존중한다.
3. **현재 구현을 최대한 깊고 상세하게 파악한 후에 계획을 세우고 개발한다.**
4. **API 호출은 반드시 테스트 후 응답 결과를 확인하며 개발한다.**

---

## 개발 3단계 워크플로우

### 1단계 — 기능 정의 (Feature Spec)
- 무엇을 만들 것인지 설명하는 Markdown 파일을 생성한다.
- 파일 경로 규칙: `docs/features/{feature-name}.md`
- 포함 내용:
  - 기능 개요 (무엇을, 왜)
  - 사용자 시나리오
  - 주요 데이터/인터페이스 개요
  - 예상 범위 및 제외 항목(Out of Scope)
- ✅ 이 단계는 AI가 자유롭게 초안 작성 가능
- ⛔ 코드 작성 금지

---

### 2단계 — 구현 계획 (Implementation Plan)
- 1단계 Spec 파일을 기반으로 구현 계획을 설명하는 Markdown 파일을 생성한다.
- 파일 경로 규칙: `docs/plans/{feature-name}-plan.md`
- 포함 내용:
  - 기술 스택 및 선택 이유
  - 파일/모듈 구조
  - 구현 순서 및 단계별 할 일 (Task List)
  - API 연동이 있는 경우: 테스트 방법 및 확인할 응답 필드 명시
  - 리스크 및 고려 사항
- ✅ 이 단계는 AI가 초안 작성 후 **주석/코멘트로 충분한 피드백을 주고받는다.**
- ⛔ 코드 작성 금지

---

### 3단계 — 구현 (Implementation)
- **개발자가 명시적으로 "구현해줘" 또는 "implement" 명령을 내린 경우에만 시작한다.**
- 구현 원칙:
  - 기존 코드/구조를 최대한 유지하며 **최소한의 변경**으로 구현
  - 한 번에 전체를 구현하지 않고, 계획의 Task 단위로 나눠 구현
  - API 호출이 포함된 경우 반드시 **테스트 먼저 실행 → 응답 확인 → 본 구현** 순서를 따름
- ✅ 구현 후 변경 요약을 간단히 정리해준다.

---

## API 개발 규칙

> 외부 API(엔카, 케이카 등) 또는 내부 API를 다룰 때 반드시 아래 절차를 따른다.

1. **테스트 호출 먼저**: 실제 URL, 파라미터, 헤더를 확인하고 테스트 요청을 실행한다.
2. **응답 결과 확인**: 응답 구조(필드명, 타입, 페이징 방식 등)를 문서화한다.
3. **이상 응답 처리 논의**: 예상과 다른 응답이 있으면 구현 전에 먼저 공유한다.
4. **본 구현**: 응답 구조가 확인된 이후에만 파싱/가공 코드를 작성한다.

```
docs/
  features/     # 1단계: 기능 정의 Spec 파일
  plans/        # 2단계: 구현 계획 파일
AGENTS.md       # 이 파일 — 협업 규칙 정의
README.md       # 프로젝트 개요
```

## 커뮤니케이션 규칙

| 상황 | AI의 행동 |
|------|-----------|
| 기능 아이디어를 제시받은 경우 | 1단계 Spec 파일 초안 작성 제안 |
| Spec 파일이 있고 계획을 요청한 경우 | 2단계 Plan 파일 초안 작성 |
| 피드백/수정 요청이 온 경우 | 해당 md 파일 수정, 코드 작성 금지 |
| "구현해줘" 명령이 온 경우 | 3단계 진입, Task 단위로 구현 시작 |
| API 관련 작업이 포함된 경우 | 테스트 먼저 실행 후 결과 공유 |
