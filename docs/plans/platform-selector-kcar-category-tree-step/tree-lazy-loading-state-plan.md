# Sub Task Plan: 트리 노드 Lazy Load 및 상태 전이

## 목표
- 노드 확장 시 하위 depth 데이터를 지연 로딩하고, 비동기 응답 역전에도 상태 일관성을 유지한다.

## 작업 범위
- 노드별 로딩 상태(`idle|loading|success|empty|error`) 관리
- depth별 API 호출 체인 연결
- 동일 노드 재확장 시 캐시 재사용 여부 정책 확정
- 요청 순서 역전(race condition) 방지 로직 적용

## 구현 순서
1. 노드별 request sequence/token 구조 추가
2. 1차 노드 확장 시 2차 목록 lazy load 연결
3. 2차/3차 동일 패턴으로 3차/4차 load 연결
4. 응답 도착 시 현재 선택/확장 상태와 일치 여부 검증 후 반영
5. 오류 발생 시 노드 단위 재시도 동작 연결

## 완료 기준
- 각 depth 노드 확장 시 필요한 하위 목록만 로드된다.
- 오래된 응답이 현재 UI 상태를 덮어쓰지 않는다.
- 노드별 오류/재시도가 독립적으로 동작한다.

## 의존성
- Task 1 노드 모델/필터 정책

## 리스크
- 캐시 정책이 과도하면 최신 데이터 반영이 지연될 수 있다.
